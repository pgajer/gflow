# gflow Split Audit SA2: Dependency Graph

Generated: 2026-06-03

Branch: `codex/gflow-split-audit-sa2`

This report starts the dependency-graph audit for the proposed `gflow` /
`geosmooth` / `gflowx` split.  It does not move source files.  The goal is to
identify the dependency closure that must be understood before extracting any
method into `geosmooth`.

## High-Level Findings

The proposed `geosmooth` set is internally coherent but depends on a small
number of cross-package services:

1. graph construction and graph distance services from `gflow`;
2. local chart and local dimension services currently marked `shared`;
3. Rcpp/native registration and compiled numerical kernels;
4. a few utility helpers currently marked `shared` or `undecided`;
5. optional external packages for specific solvers and baselines.

The largest practical split risk is not R-level function names.  It is native
code and shared local-neighborhood infrastructure.

## Pilot: `kernel.local.polynomial.cv`

The pilot file `R/kernel_local_polynomial_cv.R` has a small and favorable
dependency closure.

### Native / C++ Dependencies

The pilot uses these native implementation files:

- `src/kernel_local_polynomial_cv_rcpp.cpp`
- `src/local_pca_charts_rcpp.cpp`
- `src/local_pca_charts.cpp`
- `src/local_pca_charts.hpp`
- generated registration in `src/RcppExports.cpp` and `src/init.c`

The ambient-coordinate backend uses ANN directly:

- `src/kernel_local_polynomial_cv_rcpp.cpp` constructs `ANNkd_tree` objects.

The local-PCA chart backend uses the shared local-PCA helper:

- `src/local_pca_charts_rcpp.cpp` calls
  `gflow::compute_local_pca_chart`;
- `src/local_pca_charts.cpp` implements `compute_local_pca_chart`.

### Pilot Verdict

`kernel.local.polynomial.cv` is still the best pilot extraction candidate, but
not as a single-file move.  A clean extraction requires first factoring two
small helper contracts:

1. a local polynomial design helper currently borrowed from `R/malps.R`;
2. a local chart/dimension helper contract covering
   `.local.pca.auto.chart.dim.with.metric` and `rcpp_local_pca_chart`.

After those are explicit, the pilot can move to `geosmooth` without copying
large parts of graph infrastructure.

## Boundary Implications

### Keep in `gflow`

These services should remain in `gflow` and be imported by `geosmooth`:

- graph builders such as `create.rknn.graph`, `create.iknn.graphs`, and related
  graph repair functions;
- graph shortest paths and graph geodesic payload validation;
- extrema/hop-neighborhood utilities if they remain part of graph/Morse-Smale
  infrastructure;
- generic kNN graph construction services.

### Move to `geosmooth`

These should probably move with `geosmooth`:

- local polynomial smoother code and its C++ ambient-coordinate backend;
- MALPS/LPL/SLPLiFT/SSRHE response-model code;
- local polynomial design helpers currently embedded in `R/malps.R`;
- response-CV utilities once they are factored away from graph-only code.

### Needs SA3 Decision

These should not be moved until compiled-code and API risks are audited:

- `R/RcppExports.R`, `src/RcppExports.cpp`, and `src/init.c`;
- `src/local_pca_charts.*`;
- ANN/kNN access from `src/kernel_local_polynomial_cv_rcpp.cpp`;
- low-level linear model helpers in `src/lm.c`, `src/mlm.cpp`, and
  `src/1D_linear_models.cpp`;
- direct `.Call` symbols used by harmonic, metric-lowpass, SSRHE, RDGraph, and
  ulogit code.

## SA2 Recommendations

1. Do not physically split packages yet.
2. Before moving the pilot, factor `.malps.design.matrix` into a neutral
   `geosmooth` local-polynomial helper file.
3. Decide whether the shared local-PCA chart backend belongs in `gflow` as a
   geometry service or in `geosmooth` as smoother infrastructure.  My current
   recommendation is to keep the low-level chart backend in `gflow` for now and
   expose a stable utility contract, because SLPLiFT/LPL/MALPS use it but it is
   fundamentally geometric.
4. Treat `kernel.local.polynomial.cv` pilot extraction as a two-step operation:
   first factor helper contracts, then move the pilot method.
5. Start SA3 before any move that touches native registration.

## SA2 Exit Status

SA2 is started and the first dependency graph artifacts exist.  It is not yet a
complete split authorization.  The next useful step is **SA3: Compiled-Code
Audit**, with special attention to the pilot smoother, local-PCA chart backend,
ANN/kNN ownership, and native registration strategy.
