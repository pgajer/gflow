## Main Finding

Moving the smoother family together is feasible and cleaner than moving only
`kernel.local.polynomial.cv`, but the move must be done as a compiled-code-aware
cluster.  The cluster is not blocked by `.malps.design.matrix`, because MALPS
and kernel local polynomial would move together.  The real decisions are:

1. ownership of the local-PCA chart backend;
2. ownership of the ANN dependency used by the optimized kernel smoother;
3. regeneration of Rcpp/native registration in the new package;
4. whether SSRHE moves in the first physical extraction or follows after a
   local-PCA API is frozen.

## External and Vendored Dependencies

### ANN

The optimized kernel smoother uses ANN directly:

- `src/kernel_local_polynomial_cv_rcpp.cpp` includes `<ANN/ANN.h>`;
- it constructs `ANNkd_tree` objects;
- it calls `annAllocPts`, `annkSearch`, and `annClose`.

Current package build rules compile vendored ANN sources:

- `src/Makevars` sets `PKG_CPPFLAGS = -I. -IANN`;
- `src/Makevars` includes `ANN_SOURCES = $(wildcard ANN/*.cpp)`;
- ANN license text is in `inst/licenses/ANN-Copyright-Notice.txt`.

If `src/kernel_local_polynomial_cv_rcpp.cpp` moves to `geosmooth`, then
`geosmooth` must either:

1. vendor ANN and carry the ANN license, or
2. use a stable `gflow` kNN service instead of direct ANN access, or
3. temporarily fall back to the R backend until a shared kNN/native API is
   designed.

Option 1 is the fastest physical extraction.  Option 2 is cleaner long-term but
requires a deliberate shared native/API design.

### Eigen

The local-PCA and SSRHE code use Eigen:

- `src/local_pca_charts.*`
- `src/ssrhe_hessian_energy.cpp`

Eigen headers are vendored under `inst/include/Eigen`.  If SSRHE moves while
local PCA remains in `gflow`, `geosmooth` still needs Eigen for SSRHE's Hessian
operator.  This means `geosmooth` will likely need its own Eigen access or a
`LinkingTo`/vendoring plan regardless of the local-PCA ownership decision.

### LAPACK

The kernel smoother C++ backend uses LAPACK:

- `src/kernel_local_polynomial_cv_rcpp.cpp` calls `dpotrf` and `dpotrs` through
  `R_ext/Lapack.h`.

This is acceptable for `geosmooth`, but it must be represented in its native
build configuration.

## Local-PCA Ownership Decision

The local-PCA backend is used by:

- kernel local polynomial, when `coordinate.method = "local.pca"`;
- LPL-TF, through `.lpl.tf.local.chart`;
- SSRHE, internally in `S_ssrhe_hessian_operator`.

The code is geometric rather than method-specific.  My recommendation is:

1. keep low-level local-PCA chart construction in `gflow` for the first split;
2. expose a stable public or semi-public API from `gflow`, instead of relying
   on unexported Rcpp wrappers;
3. let `geosmooth` import that API for kernel local polynomial and LPL-TF;
4. allow SSRHE to keep using the C++ helper only if SSRHE stays in `gflow`, or
   duplicate/move the helper only if SSRHE moves immediately.

This creates one open decision: if SSRHE is included in the first extraction,
either `src/local_pca_charts.*` must move/duplicate into `geosmooth`, or
`geosmooth` must gain a supported compiled-code way to call the `gflow` local
PCA helper.  The latter is cleaner architecturally but more work.

## Recommended Physical Extraction Strategy

### Step 1: Do Not Rename Public Functions

Keep current user-facing names during extraction:

- `kernel.local.polynomial.cv`
- `fit.malps`
- `fit.lpl.tf`
- `fit.slpl.tf`
- `ssrhe.hessian.operator`

Add cleaner aliases only after `geosmooth` is installable and passing tests.

### Step 2: Create `geosmooth` With R-Only Cluster First

Move the R files for:

- kernel local polynomial;
- MALPS;
- LPL-TF;
- SLPLiFT / S-LPL-TF;
- shared R helper files needed by these methods.

Keep compiled backends disabled or imported during the first skeleton build if
necessary.  This isolates namespace/documentation/test issues from native-code
issues.

### Step 3: Add Native Backends Deliberately

Add native code in this order:

1. kernel local polynomial C++ backend;
2. local-PCA API bridge or local-PCA helper ownership decision;
3. LPL/SLPLiFT local-PCA path;
4. SSRHE Hessian operator.

This order gives early value while preventing the SSRHE/local-PCA C++ coupling
from dominating the first extraction.

### Step 4: Decide ANN Ownership

Before enabling the optimized kernel backend in `geosmooth`, choose one:

- vendor ANN in `geosmooth`;
- expose a stable kNN service from `gflow`;
- temporarily keep a slower backend until shared kNN is designed.

For speed, vendoring ANN in `geosmooth` is easiest.  For long-term package
architecture, a stable `gflow` kNN service is cleaner.

### Step 5: Move Tests With the Cluster

The first test payload should include:

- `tests/testthat/test-kernel-local-polynomial-cv.R`
- `tests/testthat/test-malps.R`
- `tests/testthat/test-lpl-tf.R`
- `tests/testthat/test-slpl-tf.R`
- `tests/testthat/test-ssrhe-hessian-energy.R`
- `tests/testthat/test-local-pca-charts.R`, only if the local-PCA API lives in
  `geosmooth`; otherwise keep it in `gflow`.

## Later `geosmooth`-Like Compiled Methods

The following methods are smoothing/regression-like but should not be included
in the first cluster move:

- harmonic extension/smoothing;
- mean-shift smoothers;
- metric graph low-pass;
- RDGraph regression;
- `ulogit` / `eigen.ulogit`.

They have their own direct `.Call` symbols and should get separate SA3-style
audits before moving.

## SA3 Verdict

The cluster move is reasonable, but only after one small API design task:

> Define a stable local-neighborhood API boundary between `gflow` and
> `geosmooth`, especially for local PCA and kNN.

After that, the cleanest practical split is:

1. `gflow` keeps graph construction, graph geodesics, kNN graph services, and
   low-level local-PCA chart construction;
2. `geosmooth` owns the smoother R APIs and method-specific native kernels;
3. `gflowx` receives older experimental/application-specific smoothers later.

The first physical split should be a **cluster extraction**, not a
single-function pilot and not a public API rename.
